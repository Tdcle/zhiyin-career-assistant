当前的数据流是这样的：
Tool(search_jobs_tool) --写入--> GlobalState(内存dict) --读取--> chat_flow.py(更新按钮)

ILIKE 性能优化

改动 1：在 _init_tables 中创建 GIN 三元组索引

Python

# 已在上一次数据库改造中加入
cur.execute("CREATE EXTENSION IF NOT EXISTS pg_trgm;")

cur.execute("""
    CREATE INDEX IF NOT EXISTS idx_jobs_title_trgm
    ON jobs USING gin (title gin_trgm_ops);
""")

原理：pg_trgm 扩展将字符串拆分为三元组（trigram），建立 GIN 倒排索引。ILIKE 查询时 PostgreSQL 会自动利用该索引，避免全表扫描。

改动 2：vector_search 的 SQL 重写为 CTE 两阶段

Python

sql = """
    WITH vector_candidates AS (
        -- 第一阶段：向量 ANN 召回，仅扫描索引，极快
        SELECT *, (embedding <=> %s::vector) AS vec_dist
        FROM jobs
        WHERE embedding IS NOT NULL
        ORDER BY embedding <=> %s::vector
        LIMIT %s                            -- 召回 top_k * 4 条
    )
    -- 第二阶段：仅在召回集（几十条）内做 ILIKE，开销可忽略
    SELECT *,
        CASE
            WHEN title ILIKE %s THEN vec_dist * 0.85    -- 标题命中，距离打85折
            WHEN summary ILIKE %s THEN vec_dist * 0.92  -- 摘要命中，距离打92折
            ELSE vec_dist
        END AS final_score
    FROM vector_candidates
    ORDER BY final_score ASC
    LIMIT %s;                               -- 最终返回 top_k 条
"""

性能对比：
维度	            原方案	                                新方案
ILIKE扫描范围	全表 (可能几万/几十万行)	                仅 CTE 结果集 (几十行)
索引利用	        无	                                    GIN trgm 索引 + ivfflat 向量索引
top_k           传入	f"LIMIT {top_k}" f-string 拼接	    LIMIT %s 参数化
加权逻辑	        距离 - 0.2 (尺度不统一)	                距离 * 0.85 (比例缩放，语义合理)

1、新增意图识别：

不要让用户的自然语言直接去查数据库！在调用检索工具之前，先让大模型把用户的 Query 结构化并扩充。

做法：在大模型的 System Prompt 或者专门的意图识别节点中，将用户的输入提取为 JSON：

用户输入：“帮我找北京的实习，最好是后端”

LLM 提取并扩充：

JSON
{
  "city": "北京",
  "job_type": "实习",
  "intent_keywords": ["后端", "服务端", "Java", "Go", "C++", "Python后台", "Node.js"],
  "semantic_query": "北京 后端开发 实习生 服务端工程师"
}
优势：通过 LLM 的先验知识，把“后端”平滑扩展为具体的技能栈，去掉了数据科学领域的 Python 噪音（因为扩充词带有“后台/服务端”限定）。


2. 向量的弱点补足：硬性条件过滤 (Metadata Pre-filtering)
（解决召回不相关数据的痛点）

向量检索最怕什么？最怕“硬条件”。用户说“只要北京的”，向量模型可能觉得上海的一个高度相似的岗位得分更高。

做法：结合第 1 步提取出的 city、experience（实习）等字段，在 SQL 查询时，把它们作为硬性 WHERE 条件，然后再算向量距离。

优化后的 SQL 概念：

SQL
WITH filtered_jobs AS (
    SELECT * FROM jobs
    WHERE city LIKE '%北京%' AND (title LIKE '%实习%' OR experience = '在校/应届')
)
SELECT *, (embedding <=> %s::vector) AS vec_dist
FROM filtered_jobs
ORDER BY vec_dist ...
优势：极大减少了计算量，100% 保证地域、薪资、年限等硬指标的绝对准确，避免大模型“胡乱推荐”。



3. 升级你的“混合检索”：PostgreSQL 原生全文检索 + RRF 融合
你目前的混合检索是用 ILIKE 做加权（vec_dist * 0.85）。这种方式比较粗糙，且 ILIKE '%前端%' 无法匹配“前端开发”。
既然你用的是 PostgreSQL，强烈建议开启它的原生全文检索（Full Text Search）。

做法：

建表时增加一个 tsvector 字段，把标题、详情分词存进去。

查询时，一路走向量库（算 Semantic Score），一路走全文检索（算 BM25/Keyword Score）。

使用 RRF (Reciprocal Rank Fusion, 倒数排序融合) 算法合并两路结果。

优势：向量负责“懂你的意思”（后端匹配到服务器），全文检索负责“精准咬字”（必须包含 Vue3）。RRF 融合算法不需要你手动拍脑袋定 0.85 的权重，而是通过排名公式自动融合，是业界公认最稳的方案。
