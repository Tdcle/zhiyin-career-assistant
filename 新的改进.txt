现状分析

当前的数据流是这样的：
Tool(search_jobs_tool) --写入--> GlobalState(内存dict) --读取--> chat_flow.py(更新按钮)

核心问题：Tool 执行完毕后，需要把结构化的职位列表传给前端 UI 更新按钮，但 Tool 的返回值是给 LLM 看的纯文本，不是给前端看的结构化数据。所以你用了一个全局字典做"暗通道"。
改进方案：利用 LangGraph State 传递，彻底去掉 GlobalState

思路：在 AgentState 中新增一个字段 search_results，Tool 执行完后数据自然留在 State 里，chat_flow.py 从 Graph 的最终 State 中读取即可。



ILIKE 性能优化

改动 1：在 _init_tables 中创建 GIN 三元组索引

Python

# 已在上一次数据库改造中加入
cur.execute("CREATE EXTENSION IF NOT EXISTS pg_trgm;")

cur.execute("""
    CREATE INDEX IF NOT EXISTS idx_jobs_title_trgm
    ON jobs USING gin (title gin_trgm_ops);
""")

原理：pg_trgm 扩展将字符串拆分为三元组（trigram），建立 GIN 倒排索引。ILIKE 查询时 PostgreSQL 会自动利用该索引，避免全表扫描。

改动 2：vector_search 的 SQL 重写为 CTE 两阶段

Python

sql = """
    WITH vector_candidates AS (
        -- 第一阶段：向量 ANN 召回，仅扫描索引，极快
        SELECT *, (embedding <=> %s::vector) AS vec_dist
        FROM jobs
        WHERE embedding IS NOT NULL
        ORDER BY embedding <=> %s::vector
        LIMIT %s                            -- 召回 top_k * 4 条
    )
    -- 第二阶段：仅在召回集（几十条）内做 ILIKE，开销可忽略
    SELECT *,
        CASE
            WHEN title ILIKE %s THEN vec_dist * 0.85    -- 标题命中，距离打85折
            WHEN summary ILIKE %s THEN vec_dist * 0.92  -- 摘要命中，距离打92折
            ELSE vec_dist
        END AS final_score
    FROM vector_candidates
    ORDER BY final_score ASC
    LIMIT %s;                               -- 最终返回 top_k 条
"""

性能对比：
维度	            原方案	                                新方案
ILIKE扫描范围	全表 (可能几万/几十万行)	                仅 CTE 结果集 (几十行)
索引利用	        无	                                    GIN trgm 索引 + ivfflat 向量索引
top_k           传入	f"LIMIT {top_k}" f-string 拼接	    LIMIT %s 参数化
加权逻辑	        距离 - 0.2 (尺度不统一)	                距离 * 0.85 (比例缩放，语义合理)
